# Куда и когда указывать BACKEND_URL, FRONTEND_URL и URL WebApp в Telegram

## Краткая схема

```
[ Telegram ]  →  открывает по ссылке  →  [ Фронт (WebApp) ]  →  запросы к API  →  [ Бэкенд ]
                     ↑                              ↑                                    ↑
              URL из BotFather              FRONTEND_URL +                    BACKEND_URL
              = адрес фронта                 config.json backendUrl
```

- **В Telegram (BotFather)** указывается **один** URL — тот, по которому открывается ваше мини-приложение (фронт).
- **BACKEND_URL** — адрес бэкенда (API). Его должны знать бот и браузер (фронт).
- **FRONTEND_URL** — адрес фронта. Его должен знать бэкенд (CORS, редиректы после оплаты).

---

## 1. Где что указывать

### 1.1. BACKEND_URL

**Где:** в файле **`.env` в корне проекта** (рядом с `backend/`).

**Что это:** Публичный URL, по которому доступен ваш **бэкенд** (FastAPI, uvicorn). То есть по этому адресу должны открываться эндпоинты вида `/health`, `/api/config`, `/api/bookings` и т.д.

**Кто использует:**
- **Бот** — все запросы к API (регистрация, брони, мойки) идут на `BACKEND_URL`.
- **CORS** — бэкенд разрешает запросы с `FRONTEND_URL`; сам `BACKEND_URL` здесь не подставляется, но фронт как раз ходит на бэкенд по своему `backendUrl` (см. ниже).

**Когда указывать:**
- **Локально без туннеля:** можно оставить `http://localhost:8000` (только для тестов с того же компьютера).
- **Доступ с телефона или из Telegram:** нужен **публичный** URL бэкенда, например:
  - отдельный туннель до порта 8000: `https://ваш-бэкенд.serveo.net` или `https://xxxx.ngrok.io`;
  - или домен сервера: `https://api.ваш-сайт.ru`.

**Пример в `.env`:**
```env
BACKEND_URL=https://abc123-backend.serveo.net
```
(замените на ваш реальный публичный URL бэкенда.)

---

### 1.2. FRONTEND_URL

**Где:** в том же **`.env` в корне проекта**.

**Что это:** Публичный URL, по которому открывается **фронт** (ваше WebApp). С этого адреса пользователь получает `index.html` и все статику (JS, CSS).

**Кто использует:**
- **Бэкенд** — CORS (разрешает запросы с этого origin) и редирект после оплаты ЮKassa: пользователя отправляют на `FRONTEND_URL/payment-success?booking_id=...`.

**Когда указывать:**
- **Локально без туннеля:** `http://localhost:5173`.
- **Доступ с телефона / из Telegram:** публичный URL фронта, например:
  - туннель до порта 5173: `https://ваш-фронт.serveo.net`;
  - или домен: `https://ваш-сайт.ru`.

**Пример в `.env`:**
```env
FRONTEND_URL=https://xyz789-front.serveo.net
```
(замените на ваш реальный публичный URL фронта.)

---

### 1.3. URL WebApp в Telegram (BotFather)

**Где:** в **BotFather** (Telegram) → ваш бот → **Bot Settings** → **Menu Button** / **Configure menu button** или в коде бота URL кнопки «Открыть мини-приложение».

**Что указывать:** Ровно тот же адрес, по которому отдаётся ваше **фронтенд-приложение** (то же, что и `FRONTEND_URL`).

То есть:
- в **BotFather** (или в коде бота) для кнопки Web App указываете: `https://ваш-фронт.serveo.net` (без `/payment-success` и без путей);
- в **`.env`** в `FRONTEND_URL` — тот же `https://ваш-фронт.serveo.net`.

Когда пользователь нажимает «Открыть мини-приложение», Telegram открывает в WebView именно этот URL. Если подставить сюда адрес бэкенда — откроется API, а не приложение, поэтому в Telegram всегда указывается **URL фронта** = **FRONTEND_URL**.

**Итог:** в Telegram указываете тот же URL, что и в `FRONTEND_URL`.

---

### 1.4. Откуда фронт узнаёт адрес бэкенда (backendUrl)

Браузер (и WebApp во вкладке Telegram) делает запросы к API не «на тот же адрес», а на отдельный домен бэкенда. Этот адрес фронт берёт из одного из двух мест.

#### Вариант A: файл `config.json` (рекомендуется для туннелей)

**Где:** в **корне фронта при раздаче**. При разработке — файл в `frontend/public/config.json` (Vite копирует его в корень при `npm run build` и отдаёт при `npm run dev`).

**Содержимое:**
```json
{"backendUrl": "https://ваш-бэкенд.serveo.net"}
```

Подставьте сюда **тот же** публичный URL, что и в `BACKEND_URL` в `.env`. Без завершающего слэша.

**Когда использовать:** когда открываете WebApp с телефона или с другого компьютера. Тогда фронт грузится с `FRONTEND_URL`, запрашивает `FRONTEND_URL/config.json`, получает `backendUrl` и все запросы к API отправляет на бэкенд. Так устраняется ошибка «Failed to fetch» при запросе моек и т.д.

#### Вариант B: переменная сборки VITE_BACKEND_URL

**Где:** при **сборке** фронта, например в `frontend/.env` или в команде:
```bash
VITE_BACKEND_URL=https://ваш-бэкенд.serveo.net npm run build
```

**Когда использовать:** если вы всегда собираете статику под один конкретный бэкенд и не хотите править `config.json` после сборки.

**Итог:** в `config.json` → `backendUrl` (или в `VITE_BACKEND_URL`) указываете тот же адрес, что и в **BACKEND_URL** в корневом `.env`.

## 2. Сводная таблица

| Переменная / настройка | Где задаётся | Что подставлять | Зачем |
|------------------------|--------------|-----------------|--------|
| **BACKEND_URL** | `.env` в корне проекта | Публичный URL бэкенда (uvicorn, порт 8000) | Бот и (косвенно) фронт ходят в API по этому адресу |
| **FRONTEND_URL** | `.env` в корне проекта | Публичный URL фронта (Vite, порт 5173) | CORS и редирект после оплаты |
| **Web App URL** (в Telegram) | BotFather или код бота | То же, что FRONTEND_URL | По этой ссылке открывается ваше мини-приложение |
| **config.json** → `backendUrl` | `frontend/public/config.json` | То же, что BACKEND_URL | Чтобы браузер/WebApp знал, куда слать запросы к API |

Правило: **в Telegram — только URL фронта. В config.json (или VITE_BACKEND_URL) — только URL бэкенда.**

## 3. Типичные сценарии

### Только локально (компьютер, без телефона)

- **BACKEND_URL** = `http://localhost:8000`
- **FRONTEND_URL** = `http://localhost:5173`
- Web App в Telegram можно не использовать или указать туннель (см. ниже).
- `config.json` можно не трогать — с localhost фронт по умолчанию может использовать тот же origin или localhost:8000.

### Разработка с телефона / WebApp из Telegram (два туннеля)

Нужны **два** публичных URL: один для фронта, один для бэкенда.

1. Запустить **фронт:** `npm run dev` (порт 5173).
2. Запустить **бэкенд:** `uvicorn ... --port 8000`.
3. Поднять **туннель на 5173** → получите URL фронта, например `https://xyz.serveo.net`.
4. Поднять **туннель на 8000** → получите URL бэкенда, например `https://abc.serveo.net`.

В **`.env`**:
```env
BACKEND_URL=https://abc.serveo.net
FRONTEND_URL=https://xyz.serveo.net
```

В **Telegram** (BotFather / кнопка Web App): `https://xyz.serveo.net`.

В **`frontend/public/config.json`**:
```json
{"backendUrl": "https://abc.serveo.net"}
```

Перезапустить фронт (или обновить страницу), чтобы подхватился `config.json`.

### Продакшен (один домен, nginx)

На одном домене можно отдавать и фронт, и API, например:
- `https://ваш-сайт.ru/` → фронт;
- `https://ваш-сайт.ru/api/` → проксирование на uvicorn.

Тогда:
- **FRONTEND_URL** = `https://ваш-сайт.ru`
- **BACKEND_URL** = `https://ваш-сайт.ru` (или тот же домен; запросы к `/api/...` идут на тот же origin).
- В Telegram: `https://ваш-сайт.ru`.
- В `config.json`: `{"backendUrl": "https://ваш-сайт.ru"}` (или можно не использовать, если фронт ходит на тот же origin).

## 4. Ошибка 502 при открытии WebApp (страница не работает)

Сообщение вида **«Сейчас эта страница не работает … не удается обработать этот запрос. HTTP ERROR 502»** при открытии ссылки типа `https://....serveousercontent.com` означает следующее.

**502 = Bad Gateway:** сервер (туннель), который получил запрос, сам не смог получить ответ от того, на кого он перенаправляет трафик (вашего локального процесса).

То есть:
- Запрос пришёл на туннель (например, serveousercontent.com).
- Туннель пытается связаться с вашим приложением (например, с `localhost:5173` для фронта).
- Ответа нет или соединение не установилось → 502.

### Что проверить по шагам

1. **Фронт запущен и слушает порт**
   - Должна быть запущена команда в папке `frontend`: `npm run dev`.
   - В консоли обычно: `Local: http://localhost:5173/`. Порт должен быть именно тот, на который смотрит туннель (чаще всего 5173 для фронта).

2. **Туннель смотрит на правильный порт и хост**
   - Туннель для **WebApp** должен быть настроен на тот хост и порт, где крутится фронт (например, `127.0.0.1:5173` или `localhost:5173`).
   - Если туннель был поднят до порта 8000 (бэкенд), то при открытии этой ссылки в Telegram откроется не приложение, а API — и может быть 502 или страница с JSON/ошибкой.
   - Итог: URL, который вы вставили в Telegram (и в `FRONTEND_URL`), должен быть туннелем именно до **фронта (5173)**.

3. **Туннель не «отвалился»**
   - Многие туннели (Serveo, бесплатный ngrok и т.п.) перестают работать после перезапуска или через какое-то время. Тогда старый URL даёт 502.
   - Нужно заново поднять туннель, получить **новый** URL и обновить:
     - **FRONTEND_URL** в `.env`,
     - URL кнопки Web App в BotFather (или в коде бота),
     - при двух туннелях — отдельно проверить туннель для бэкенда и **BACKEND_URL** / `config.json` → `backendUrl`.

4. **Один туннель на два сервиса**
   - Если у вас **один** туннель и он смотрит только на один порт (например, 5173), то по одному URL доступен только фронт. Бэкенд по этому же URL недоступен → запросы к `/api/...` на этот URL дадут 404 или 502.
   - Решение: либо **два туннеля** (один на 5173 = FRONTEND_URL и Telegram, второй на 8000 = BACKEND_URL и config.json), либо один домен с прокси (nginx), который и раздаёт фронт, и проксирует `/api` на бэкенд.

### Чек-лист при 502

- [ ] В папке `frontend` выполняется `npm run dev`, в логах есть `localhost:5173`.
- [ ] Туннель для ссылки из Telegram поднят **на порт 5173** (фронт).
- [ ] В Telegram (BotFather / кнопка) указан **актуальный** URL этого туннеля (тот же, что и FRONTEND_URL).
- [ ] После смены URL туннеля обновлены FRONTEND_URL в `.env` и ссылка в Telegram.
- [ ] Для запросов к API с телефона: есть второй туннель на 8000, в `.env` указан BACKEND_URL, в `frontend/public/config.json` — `backendUrl` с тем же URL.

После этого при открытии WebApp из Telegram должна открываться ваша React-страница, а не 502.

## 5. Бот получает 502 при запросах к API (GET /api/auth/me и т.д.)

Если в логах бота видно **«HTTP/1.1 502 Bad Gateway»** при запросах к `BACKEND_URL` (например, `/api/auth/me`), это значит: **запрос доходит до туннеля, но туннель не может получить ответ от вашего бэкенда.**

### Частая причина: один туннель на два сервиса

В **BACKEND_URL** должен быть адрес туннеля, который проброшен **именно на порт 8000** (uvicorn). Если вы подняли **один** туннель и указали его и в `FRONTEND_URL`, и в `BACKEND_URL`, то:

- Туннель чаще всего смотрит на **один** порт (например, 5173).
- Запросы бота идут на этот же URL → туннель отдаёт их на 5173 (Vite). У Vite нет маршрута `/api/auth/me` → туннель или сервер возвращает **502**.

**Что сделать:**

1. Поднять **два туннеля:**
   - Туннель 1: порт **5173** (фронт) → его URL только в **FRONTEND_URL** и в Telegram (Web App).
   - Туннель 2: порт **8000** (бэкенд) → его URL только в **BACKEND_URL** (и в `config.json` → `backendUrl` для фронта).
2. В **.env** указать:
   - `BACKEND_URL=https://ваш-туннель-на-8000.serveousercontent.com`
   - `FRONTEND_URL=https://ваш-туннель-на-5173.serveousercontent.com`
3. **Перед запуском бота** запустить бэкенд:  
   `uvicorn backend.app.main:app --reload --host 0.0.0.0 --port 8000`  
   и убедиться, что туннель на 8000 запущен и в BROWSER/Postman по адресу `BACKEND_URL/health` приходит ответ `{"status":"ok"}`.

При старте бот проверяет доступность бэкенда по `BACKEND_URL/health`. Если бэкенд недоступен или вернул не 200, в лог пишется предупреждение с подсказкой проверить туннель и порт 8000.
