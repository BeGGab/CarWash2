# Как научиться писать такие проекты: с чего начать и поэтапная разработка

Руководство описывает, с чего начать и в каком порядке создавать проект уровня CarWash: **Telegram-бот + REST API (FastAPI) + Mini App (React) + платежи + админка**. Подойдёт для агрегаторов, записей, маркетплейсов и похожих сервисов.

---

## С чего начать

### 1. Чётко сформулировать идею

Ответьте на вопросы:

- **Что делает сервис?** (например: поиск автомоек, запись на слот, предоплата.)
- **Кто пользователи?** (клиент, владелец точки, системный админ.)
- **Как они заходят?** (только через Telegram, или ещё сайт/приложение.)
- **Какие действия нужны?** (регистрация, поиск, бронь, оплата, отмена, уведомления.)

Запишите сценарии в виде коротких шагов: «Пользователь нажимает „Отправить геолокацию“ → бот шлёт запрос в API → API возвращает список моек рядом».

### 2. Выбрать стек

Для проектов «бот + веб» удобна такая связка:

| Часть        | Технология              | Зачем |
|-------------|--------------------------|-------|
| API         | FastAPI                  | Async, автодокументация, типы, легко связать с ботом и фронтом |
| БД          | PostgreSQL + SQLAlchemy (async) | Надёжность, JSON, полнотекстный поиск, миграции через Alembic |
| Миграции    | Alembic                  | Версионирование схемы, безопасные обновления на проде |
| Бот         | aiogram 3                | Async, удобные хендлеры, Web App, webhooks |
| Фронт (Mini App) | React + Vite        | Быстрая сборка, удобно встраивать в Telegram WebView |
| Платежи     | ЮKassa (или аналог)      | Приём оплаты по ссылке + webhook для подтверждения |
| Конфиг      | .env + pydantic-settings | Один источник правды для всех частей приложения |

Один репозиторий: корень — бэкенд и бот, папка `frontend/` — Mini App. Так проще деплой и общая логика.

### 3. Спроектировать данные и API

Набросайте **сущности** и связи:

- Пользователь (telegram_id, телефон, роль).
- Сущности предметной области (например: автомойка, слот, услуга, бронь).
- Платёж (сумма, статус, связь с бронь/заказом).

Потом — **ключевые операции**: регистрация, поиск, создание брони, создание платежа, webhook от платёжки, отмена, список «мои брони». Для каждой операции решите: это метод бота, метод API или оба (бот вызывает API).

Имеет смысл нарисовать схему БД (таблицы и связи) и список эндпоинтов (например в виде таблицы: метод, путь, кто вызывает). Детали можно уточнять по ходу, но база и основные сценарии должны быть ясны до кода.

---

## Этап 1: Каркас бэкенда и база данных

**Цель:** поднять FastAPI, подключить БД, завести модели и первую миграцию.

1. Создайте проект (папки `backend/app/`, корень с `requirements.txt`, `.env`).
2. **Конфиг:** один модуль настроек (pydantic-settings), все переменные из `.env`, без дефолтов-секретов в коде. Переменные: `DATABASE_URL`, позже `TELEGRAM_BOT_TOKEN`, `FRONTEND_URL` и т.д.
3. **Подключение к БД:** SQLAlchemy async engine + session factory, зависимость `get_async_session()` для FastAPI.
4. **Модели:** сначала только то, без чего нельзя обойтись (например User). Описывайте поля, связи, enum’ы статусов.
5. **Alembic:** инициализация, `env.py` с чтением URL из настроек (из .env). Первая миграция по моделям. При старте приложения в lifespan вызывайте `alembic upgrade head`, а не `create_all`.
6. **Точка входа FastAPI:** `main.py` с lifespan (сначала только миграции), CORS, роутеры. Эндпоинт `/health` для проверки.

Итог этапа: по запросу к API приложение поднимается, миграции применяются, с БД можно работать через сессии. Никакого бота и фронта пока.

---

## Этап 2: Авторизация и пользователи

**Цель:** понимать, «кто сейчас пользуется сервисом» (по Telegram или по токену).

1. **Регистрация/идентификация:** эндпоинт вида `POST /api/auth/register` с телом (telegram_id, phone, full_name). Создаётся или обновляется пользователь, возвращается роль (user, carwash_admin, system_admin).
2. **Проверка пользователя:** эндпоинт `GET /api/auth/me?telegram_id=...` (или заголовок/токен), возвращает роль и базовые данные. Им будут пользоваться бот и фронт.
3. **Роли:** в модели User — поле role (enum). Системных админов задаёте списком ID в .env; админов точек позже назначаете через отдельный сценарий.

Итог: бэкенд умеет «зарегистрировать» пользователя по telegram_id и сказать, кто он. Бот и Mini App будут передавать telegram_id при каждом важном действии.

---

## Этап 3: Предметная область (мойки, слоты, брони)

**Цель:** реализовать ядро бизнес-логики только в API.

1. **Модели:** CarWash (название, адрес, координаты, владелец), слоты/расписание (или упрощённо — дата + время), услуги, Booking (пользователь, мойка, слот, статус: pending_payment, paid, cancelled, completed).
2. **Эндпоинты для клиента:** например:
   - поиск по геолокации: `GET /api/carwashes/nearby?lat=...&lon=...&radius_km=...`;
   - слоты мойки: `GET /api/carwashes/{id}/slots` или аналог;
   - создание брони: `POST /api/bookings` (telegram_id, слот, при необходимости услуги);
   - мои брони: `GET /api/bookings/me?telegram_id=...`;
   - отмена: `POST /api/bookings/{id}/cancel?telegram_id=...`.
3. **Валидация:** не создавать бронь на занятый слот, проверять права по telegram_id. Всё через сервисный слой или прямо в роутерах — как вам удобнее, главное чтобы логика была в одном месте (бэкенд).

Итог: API полностью закрывает сценарии «найти → забронировать → отменить». Без бота и без оплаты уже можно тестировать через Swagger или curl.

---

## Этап 4: Telegram-бот

**Цель:** пользователь общается с тем же API через бота.

1. **Один процесс с API:** бот не пишет в БД сам — только вызывает API по HTTP (httpx). Так не дублируется логика и проще поддерживать.
2. **Стартовый сценарий:** команда /start → запрос к `GET /api/auth/me` → если пользователя нет, запрос контакта и `POST /api/auth/register` → показ меню по роли.
3. **Меню:** кнопки под ролью (клиент: геолокация, «Открыть Mini App», «Мои брони»; админ точки: кабинет, регистрация точки и т.д.). Клавиатуры формируйте в боте, данные — из API.
4. **Web App:** кнопка «Открыть мини-приложение» с `WebAppInfo(url=FRONTEND_URL)`. Фронт потом будет открываться в Telegram.
5. **Webhook вместо long polling:** в продакшене — один процесс: только FastAPI. Роутер `POST /api/telegram/webhook` принимает тело от Telegram, парсит Update и передаёт в `dp.feed_webhook_update(bot, update)`. В lifespan при старте — `bot.set_webhook(BACKEND_URL + "/api/telegram/webhook")`, при остановке — `delete_webhook`. Локально для отладки можно оставить запуск бота через `python -m bot.bot` (long polling).

Итог: пользователь может зайти в бота, зарегистрироваться, отправить геолокацию и получить ответ из API, открыть Mini App. Всё через один бэкенд.

---

## Этап 5: Mini App (фронт)

**Цель:** тот же функционал, что и в боте, но в удобном веб-интерфейсе внутри Telegram.

1. **Сборка:** Vite + React (или другой фреймворк), сборка в `dist/`. Для Telegram важен HTTPS и возможность задать URL приложения в BotFather.
2. **Идентификация:** во фронте получаете `telegram_id` из `Telegram.WebApp.initDataUnsafe.user.id` и передаёте в каждый запрос к API (query или body).
3. **Экраны:** главная (поиск по геолокации или по умолчанию), карточка мойки, выбор слота, подтверждение брони, оплата (редирект на платёжную систему), «Мои брони».
4. **Конфиг:** URL бэкенда не зашивать в код. Либо `config.json` в `public/`, который при деплое заполняется из переменной окружения, либо одна переменная при сборке (например `VITE_BACKEND_URL`).
5. **CORS и FRONTEND_URL:** в бэкенде в CORS и в редиректах после оплаты использовать один и тот же FRONTEND_URL из .env.

Итог: пользователь из бота нажимает «Открыть мини-приложение» и попадает во фронт; все действия идут через ваш API с тем же telegram_id.

---

## Этап 6: Платежи

**Цель:** принять оплату (например предоплату за бронь) и обновить статус в БД.

1. **Модель:** Payment (внешний id платежа, сумма, статус, связь с бронь/заказом).
2. **Создание платежа:** эндпоинт `POST /api/payments/create` (booking_id, сумма и т.д.) → запрос к API платёжной системы → возврат ссылки на оплату и `confirmation_url` с переходом на ваш фронт (например `/payment-success?booking_id=...`). В .env храните ключи и FRONTEND_URL.
3. **Webhook:** платёжная система вызывает ваш `POST /api/payments/yookassa/webhook` с телом о статусе. Проверяйте подпись/секрет, обновляйте статус платежа и связанной брони (например на «оплачено»). Ответ отдавайте быстро (200), тяжёлую логику при необходимости в фоне.
4. **Возвраты:** при отмене брони — вызов API возврата у платёжной системы и обновление статусов. Условия (например «не позднее чем за N часов») держать в конфиге.

Итог: пользователь переходит по ссылке, платит, после успеха попадает на ваш фронт; бэкенд по webhook помечает бронь оплаченной.

---

## Этап 7: Админка и доп. функции

**Цель:** владелец точки и системный админ могут управлять данными.

1. **Админ точки (carwash_admin):** эндпоинты под префиксом `/api/admin/` с проверкой `telegram_id` и роли: создание/редактирование моек, управление услугами, список броней, завершение по QR и т.д. Во фронте — отдельная страница/раздел (например `/admin-carwash.html`) с тем же способом авторизации (telegram_id из WebApp).
2. **Системный админ:** эндпоинты `/api/system/` только для ID из списка в .env: модерация моек, статистика, назначение админов точек. Отдельная страница `/system-admin.html`.
3. **Уведомления:** при новой брони — отправка сообщения владельцу в Telegram (тот же бот, вызов `bot.send_message` из бэкенда по telegram_id). Очередь или фоновые задачи при росте нагрузки.
4. **Напоминания:** планировщик (APScheduler или Celery) — задачи «за N часов до слота отправить напоминание»; при отмене брони — отмена задачи.

Итог: полный цикл для клиента и для владельца/системы, без дублирования логики между ботом и фронтом.

---

## Этап 8: Деплой и эксплуатация

1. **Один процесс в продакшене:** один процесс uvicorn; в нём же — webhook бота, миграции при старте, планировщик. Отдельно — фронт (статический хостинг или свой сервер).
2. **Переменные окружения:** все секреты и URL только в .env на сервере (или в настройках облака). Никаких дефолтов с реальными ключами в коде.
3. **Миграции:** только через Alembic. На деплое при старте приложения выполняется `alembic upgrade head` (у вас — внутри lifespan).
4. **Документация:** README со структурой проекта, примером .env и ссылками на инструкции по деплою (Python/Node и Docker). Отдельный файл с поэтапным планом (как этот) помогает и при обучении, и при онбординге.

## Порядок в итоге

1. Идея и сценарии → стек → схема данных и список эндпоинтов.  
2. Каркас бэкенда + БД + Alembic + конфиг из .env.  
3. Авторизация/пользователи (API).  
4. Ядро предметной области (мойки, слоты, брони) в API.  
5. Бот: /start, меню, вызовы API, кнопка Mini App, затем переход на webhook.  
6. Mini App: экраны, вызовы API с telegram_id, конфиг бэкенда.  
7. Платежи: создание, webhook, возвраты.  
8. Админка (API + страницы во фронте), уведомления, напоминания.  
9. Деплой: один процесс бэкенда, статика фронта, переменные и миграции.

Такой порядок даёт стабильную основу: сначала данные и API, потом бот и фронт как «клиенты» этого API, затем деньги и админка. Так проще и учиться писать такие проекты, и поддерживать их потом.
